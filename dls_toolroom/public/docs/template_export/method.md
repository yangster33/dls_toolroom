# 法

批量文档生成的实现遵循一套清晰的流水线方法论，其核心是将"数据准备—模板设计—映射渲染—结果交付"拆解为若干独立且可校验的阶段。

## 步骤一：准备数据源

将待填充的信息整理为表格结构，每一行代表一个独立的输出单元（如一份合同、一张证书、一封信函），每一列代表一个可被模板引用的字段。数据的列名（表头）将作为模板占位符的标识符。数据源可以是电子表格或逗号分隔的纯文本格式，但必须保证第一行为字段名或明确标记出字段行所在的位置。

**为什么要表格化**：表格的"行=记录、列=属性"模型是信息系统中最为通用且无歧义的数据组织方式，它能被几乎所有数据处理工具理解，也为后续的逐行迭代提供了天然边界。

## 步骤二：设计文档模板

创建一个目标格式的文档文件，在需要插入数据的位置放置占位符。占位符的语法通常采用花括号包裹字段名的方式（如 `{姓名}`、`{日期}`），使其在视觉上醒目且易于通过程序识别和替换。模板的版式、字体、颜色、表格、图片等静态元素保持不动，只有占位符所在位置会被替换为实际数据值。

**为什么要占位符约定**：统一的占位符语法使得模板本身成为一份"自文档化"的设计稿——任何拿到模板的人都能通过阅读占位符理解哪些位置是动态数据，而程序也能精确地定位和替换这些位置。

## 步骤三：配置数据起始行

在读取数据源时，需要指定从第几行开始作为有效数据。通常第一行为列名（字段名），实际数据从第二行开始。但数据源可能包含多行说明、注释或空行，因此需要提供灵活的行偏移配置，确保跳过无效内容，只处理真正的数据行。

**为什么要跳过行**：现实中的数据文件往往不是"纯数据"，而是混合了元信息（制表人、日期、说明文字等）。强行要求用户提供无杂质的纯数据文件会增加使用门槛，而跳过行的机制让工具能适应各种"不够干净"但实际存在的输入。

## 步骤四：建立字段映射

程序读取数据源后，自动提取列名作为可用字段列表。用户上传模板后，可以确认数据列名与模板占位符之间的对应关系。字段名应保持一致（包括大小写、中英文），这是数据正确填入模板的前提。

## 步骤五：定义输出命名规则

每个生成的文件需要一个唯一的文件名。允许用户使用包含占位符的命名模板（如 `{姓名}-{日期}`），程序会根据每行数据自动替换占位符生成实际文件名。同时需要处理一种常见情况：多行数据可能产生相同的文件名，此时需要引入去重策略——在文件名末尾追加递增序号，或以前缀形式添加行号。

**为什么要去重**：在批量处理场景中，同名冲突是高频问题（例如多个人同名同姓）。如果没有去重机制，后续生成的文件会覆盖前面的文件，导致数据丢失。

## 步骤六：逐行渲染

这是核心执行阶段。对数据源中的每一行执行以下操作：
1. 加载模板文档的原始结构；
2. 将当前行的字段值注入模板中对应占位符的位置；
3. 处理特殊数据类型（如日期格式转换、数值精度控制、文本截断、大小写转换等数据变换需求）；
4. 生成一份完整的输出文档。

每一行的渲染是独立的，互不干扰，这保证了即使某一行处理异常也不会影响其他行的正常输出。

## 步骤七：打包交付

所有行渲染完毕后，将结果文档交付给用户。若只有一份输出文档，则直接提供单文件下载；若有多份，则将它们打包为一个压缩归档文件一次性下载，避免用户逐个下载的繁琐操作。

## 设计原则

- **数据与版式分离**：数据源只负责内容，模板只负责形式，二者通过占位符约定松耦合地连接。任何一方的修改不会破坏另一方。
- **容错优先**：当模板中引用的字段在数据中不存在时，不中断整个流程，而是以空值替代并继续处理后续行。单行失败不应导致批量任务整体失败。
- **就近校验**：在流程的各个关键节点（数据读取、模板解析、文件名生成）尽早发现并报告问题，避免用户在完成全部操作后才发现上游步骤的错误。
